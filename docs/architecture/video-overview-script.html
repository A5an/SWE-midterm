Project Overview Video Script

1) the team responsibilities and contributions (~1-2min) 2) describe the most important design decisions using the diagrams and markdown deliverables (~5-6min) and 3) a proof of concept video with the system in action (~2-4min).

Target Length About 6 minutes
Speaking Style Slow, steady, plain English
Note Underlined lines mark design choices and the reason for them.
Read the “On screen” line as your navigation cue. Read the “Say” line out loud. Pause very briefly between sections.

0:00 - 0:30

1. Opening and Report Structure

On screen: Show the top of the combined HTML document, the title, and the table of contents.

Say: In this video, I will give a simple overview of our collaborative document editor with an AI writing assistant. The report has two parts. Part 1 explains what the system must do. Part 2 explains how we plan to build it. I will follow that order.

We split the report into requirements first and architecture second because it is easier to judge the design after the goals are clear.

0:30 - 1:15

2. Project Scope and Stakeholders

On screen: Scroll to Part 1, then show the scope and stakeholder analysis tables.

Say: The project is a web app for shared documents. Multiple users can edit at the same time, and the AI can help rewrite, summarize, translate, and restructure text. The stakeholder analysis matters because it shows which people shape the system and which features come from them.

Say: Workspace owners and admins drive permission controls, AI usage limits, cost reports, and admin rules. These features help control access, sharing, and spending.

Say: Security and compliance officers drive encryption, which means protected data, data minimization, which means sending only what is needed, retention limits, which means deleting old records on time, and audit logs, which means a record of important actions.

Say: Product and business stakeholders drive speed targets, simple screens, clear feature priority, and usage tracking so the team knows what people use and what the product costs.

Say: Support teams drive clear error messages, request IDs, which are case reference numbers, and action history for users and AI. These features make incidents easier to explain.

Say: Operations teams drive uptime targets, health checks, alerts, graceful failure, which means one problem should not break the whole system, and automatic scaling when load grows.

Say: The outside AI provider drives rate limits, backup behavior when the provider is slow or down, prompt rules, which are rules for how requests are written, and contract rules for safe data handling.

Stakeholder analysis directly shapes the feature list, so each control in the system exists for a clear human reason, not just for technical style.

1:15 - 1:45

3. Functional Requirements

On screen: Move through the four functional requirement sections: collaboration, AI assistant, document management, and user management.

Say: The requirements are grouped into four areas: collaboration, AI, documents, and users. Collaboration covers live editing, presence, conflicts, and offline sync. The AI assistant runs only when asked and returns suggestions. Document management covers creating, saving, versioning, sharing, and exporting. User management covers login, permissions, sessions, and AI policy checks.

AI output is treated as a suggestion because users need control over what changes enter the final document.

1:45 - 2:10

4. Non-Functional Requirements

On screen: Show the latency, scalability, availability, security, and usability tables.

Say: The next part explains quality targets. We set clear limits for edit delay, document load time, and AI response start time. We also define targets for scale, availability, security, and usability. These numbers matter because they turn broad goals like fast or secure into checks we can test.

2:10 - 2:35

5. User Stories and Traceability

On screen: Show the user stories, then the traceability tables.

Say: After that, the report gives user stories, such as syncing offline edits, reviewing AI history, or sharing a document in read-only mode. The traceability section links each story to requirements and then to architecture components. This shows that each feature has both a user reason and a technical plan.

2:35 - 3:15

6. C4 Architecture Overview

On screen: Move through the System Context, Container, and Collaboration Component diagrams.

Say: Part 2 starts with architecture. The first diagram shows the system boundary and outside services. The second breaks the platform into its main containers: web client, API, collaboration service, AI worker, and data stores. The third zooms into the collaboration service and shows sessions, merge logic, replay after reconnect, presence, and checkpoints.

We separate the collaboration service from the AI worker because live typing must stay fast, while AI work can be slower and run in the background.

3:15 - 4:00

7. AI Integration and API Design

On screen: Show the AI integration section, then the API style and endpoint tables.

Say: The AI design is careful and limited on purpose. By default, the system sends only the selected text and a small amount of nearby context to the AI provider. That reduces privacy risk, lowers cost, and makes responses faster. AI results appear as tracked suggestions with accept, reject, edit, and partial accept actions. The API then uses REST for normal data, WebSocket for live editing, and SSE for AI status updates.

We use selection-first AI context because it lowers cost and protects private document content.

We use REST, WebSocket, and SSE together because one protocol does not fit normal data actions, live editing, and long AI progress updates equally well.

4:00 - 4:35

8. Security, Authorization, and Communication

On screen: Show the authentication and authorization section and the communication model section.

Say: Security and access control are central in this project. Owners, editors, commenters, and viewers do not have the same powers. AI use is also limited by role and policy. In the communication model, the document first loads through the API, then the client requests a collaboration session, and only after that does the live editing channel begin. This keeps access checks clear before live editing starts.

Permission checks happen before AI requests and before collaboration sessions because sensitive content should not move through the system until access is confirmed.

4:35 - 5:10

9. Code Structure and Data Model

On screen: Show the repository structure diagram, directory layout, testing structure, and data model diagram.

Say: The report then shows how the codebase is organized. Separate apps are planned for the web client, API, collaboration service, and AI worker. Shared packages hold contracts, editor logic, config, and test utilities. The data model shows how users, documents, versions, AI requests, suggestions, permissions, and audit records connect. It also shows that version history and AI history are saved, not hidden.

Shared packages are used so request types, permission rules, and document structures stay consistent across services.

5:10 - 6:00

10. Architecture Decisions and Closing

On screen: Show the ADR section and then scroll back slightly to the report summary area.

Say: The last part records the key design decisions. We chose CRDT-based collaboration so users can work at the same time and still recover well after disconnects. We chose asynchronous AI suggestions so users stay in control. We chose saved versions with revert-as-new-head so history is never destroyed. And we chose different protocols for different kinds of work. In short, this project balances live teamwork, AI assistance, safety, and clear structure.

Revert creates a new version instead of overwriting the current one because the team needs a full history and a clear audit trail.
